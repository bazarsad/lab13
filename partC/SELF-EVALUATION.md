## 1. Хэрэв шалгалт өнөөдөр болбол би энэ кодыг өөрөө бичиж чадах уу?

**Хэсэгчлэн.**

**Чадах хэсэг:**
- Express router, middleware бүтэц — энэ талыг урьд нь ажиллаж байсан
- Input validation logic — `if (!title || title.trim() === '')` шиг шалгалтуудыг өөрөө бичнэ
- HTTP status code-уудыг зөв ашиглах (400, 404, 201, 200)
- SQLite parameterized query-ийн зарчим ойлгосон

**Дахин судлах хэрэгтэй:**
- `better-sqlite3`-ийн exact API (`.prepare().all()` vs `.run()` vs `.get()` хэзээ аль нь) — мэдэж байна ч алдаа гаргаж болно
- Dynamic WHERE clause бүтэц — логик ойлгосон, syntax-г verify хийх хэрэгтэй
- Jest `supertest` integration test-ийн setup (port binding, `afterAll` cleanup) — бага алдаатай бичиж болох ч

**Дүгнэлт:** Архитектурын шийдвэрийг, яагаад ийм бүтэц авсанг тайлбарлаж чадна. Кодыг мөр мөрөөр тайлбарлаж чадна. Харин scratch-аас бичихэд цаг илүү зарцуулж, жижиг syntax алдаа гаргаж болно.

## 2. Дахин хийнэ гэвэл юуг өөрчлөх вэ?

**1. TDD (Test-Driven Development) хийх**  
Кодоос өмнө тест бичих — энэ удаа кодоо бичиж, дараа нь тест бичсэн нь зарим тохиолдолд logic-г сайжруулах боломжийг хаасан. Тест эхлээд байвал validation edge case-уудыг эрт олох байсан.

**2. AI session log-г real-time хийх**  
Зарим session-ы дарааллыг сүүлд бичсэн учир нарийн алдаануудыг мартсан байж болно. Bunsh дараа нь бичихээс илүүтэй chat дуусаагүй байхад тэмдэглэх хэрэгтэй.

**3. CLAUDE.md-г эхлээд бүрэн бичих**  
CLAUDE.md-г build явцад өөрчилсөн нь бага chaotic болсон. Part A-д л бүрэн бичиж, Part B-д зөвхөн нэмэлт засвар хийх байсан.

**4. Feature branch ашиглах**  
`main` дээр шууд commit хийсэн — feature branch ашиглавал Git history илүү цэвэр, реалист байх байсан.

## 3. Энэ туршлагаас юу сурсан бэ?

**AI-ийн хүч болон хязгаарыг ойлгосон.**  
AI нь boilerplate, scaffold, pattern-уудыг маш хурдан гаргадаг. Харин domain-specific нарийн зүйл (library-ийн exact API, security edge case) дээр алдаа гаргадаг. "Verify, don't trust" зарчим практикт яаж ажилладагийг мэдрэлийн түвшинд ойлгосон.

**Kодыг унших нь бичихээс чухал.**  
AI бичсэн кодыг ажиллуулж, pass болсон гэж хаахаас илүүтэй мөр бүрийг унших нь архитектурыг сурах хамгийн сайн арга болсон. Ойлгоогүй мөрийг "дараа болно" гэж орхиход тестийн алдааны шалтгааныг тайлбарлаж чадаагүй.

**AI-тай ажиллах нь skill биш, workflow.**  
Сурах зүйл нь "AI-аас сайн prompt хийх" биш — "AI-ийн гаралтыг шалгах, итгэж болох хэсгийг ялгах, засварлах" юм. Энэ workflow нь кодчилолын ур чадвараас хамааралтай — AI зөвхөн мэдэж байгаа хүний гарт хэрэгтэй.
